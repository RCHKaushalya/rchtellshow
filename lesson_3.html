Lesson 3: Arithmetic, Logic, and the ASCII Conversion Story

Series: NASM Learning | Platform: Ubuntu/Linux x86-64

In the first two lessons, we learned how to set up our environment, interface with the Linux kernel, move bytes around, and peek inside registers. But computers aren't just filing cabinets; they are calculators. In Lesson 3, my mentor and I dove into the core instructions that allow the CPU to perform calculations: arithmetic and bitwise logic. Along the way, we demystified the zero flag, discovered some quirky register constraints for division, and hit a profound realization about what numbers actually look like to a computer.

1. The Core Arithmetic Instructions

The CPU has dedicated instructions for basic math. The most fundamental ones operate directly on registers in place:

Instruction What it does C/JS Equivalent
add dst, src Adds src to dst, saving the result in dst dst += src
sub dst, src Subtracts src from dst, saving the result in dst dst -= src
inc dst Increments dst by 1 dst++
dec dst Decrements dst by 1 dst--
imul dst, src Multiplies dst by src, saving the result in dst dst *= src

Note that there is no separate "result" register for these operations—the answer overwrites the destination (the first operand).

2. The Hidden Watcher: rflags

Here is something standard high-level language tutorials completely gloss over: every arithmetic instruction silently updates a special CPU register called rflags.

You never write to rflags directly. Instead, the CPU automatically sets or clears individual bits (flags) in this register depending on the result of the calculation. Two flags are crucial for logic and loops:

  • ZF (Zero Flag): Set to 1 if the result of the operation was exactly zero. Otherwise, it is 0.
  • SF (Sign Flag): Set to 1 if the result was negative. Otherwise, it is 0.

3. Reason Before You Code: The Step-by-Step Test

Before jumping into assembly, my mentor wanted to test my ability to track register states mentally. He gave me this sequence of instructions:

mov rax, 10
mov rbx, 3
add rax, rbx
sub rax, 1
inc rax
imul rax, rbx

Mentor: "Before writing anything, tell me what you think the value in rax will be after each line, step by step."

Me: "Let's trace it through:

  • mov rax, 10 → rax = 10
  • mov rbx, 3 → rax = 10 (and rbx = 3)
  • add rax, rbx → rax = 13 (which is 10 + 3)
  • sub rax, 1 → rax = 12 (which is 13 - 1)
  • inc rax → rax = 13 (incrementing 12)
  • imul rax, rbx → rax = 39 (which is 13 * 3)"

Mentor: "Perfect. Every single value correct. Now let's verify it using the actual program and GDB."

I wrote the code into a file called arithmetic.asm:

section .text
    global _start

_start:
    mov rax, 10
    mov rbx, 3
    add rax, rbx
    sub rax, 1
    inc rax
    imul rax, rbx

    mov rax, 60
    mov rdi, 0
    syscall

I compiled and linked it with debug flags, then fired up GDB:

nasm -f elf64 -g -F dwarf arithmetic.asm -o arithmetic.o
ld arithmetic.o -o arithmetic
gdb ./arithmetic

Inside GDB, I loaded the registers layout and stepped through using si (step instruction):

(gdb) break _start
(gdb) run
(gdb) layout regs

As expected, rax updated exactly as we predicted, ending at 39 (or 0x27 in hex).

4. The Extra Experiment: Catching the Zero Flag

While stepping through the instructions, my mentor told me to look closely at the eflags row in the GDB registers panel. After sub rax, 1 (where rax becomes 12), the eflags output showed 0x202 [IF] (or 0x206 [PE_IF] depending on the environment). The Zero Flag (ZF) was absent.

Then, he asked me to add one line to the code right before the exit syscall: sub rax, 39. Since rax was 39, subtracting 39 would reduce it to 0. I stepped to that instruction in GDB and watched the flags closely.

Me: "Aha! After running sub rax, 39, the register rax becomes 0, and the flags register immediately changed from eflags 0x202 [IF] to eflags 0x246 [PE ZF_IF]. There is a new ZF flag listed!"

Mentor: "Excellent observation. You caught exactly what matters. When rax hit 0, the CPU set the Zero Flag (ZF) automatically. You didn't write an instruction to do that—it is a hardware-level reaction. This flag is the bedrock of control flow. Next week, we will write jumps that check this exact flag to make decisions. That is how loop boundaries and if-else structures are built."

5. Division: The Quirky Exception

While addition, subtraction, and multiplication are simple in-place operations, division (div and idiv) works differently and is much more restricted. You cannot divide two arbitrary registers.

Instead, the idiv instruction uses a fixed combination of registers:

mov rax, 39      ; Dividend (lower 64 bits) goes in rax
mov rbx, 3       ; Divisor (can be any general register)
xor rdx, rdx     ; IMPORTANT: Clear rdx first!
idiv rbx         ; rax = quotient (13), rdx = remainder (0)

There are two major gotchas here:

  1. The 128-bit Dividend: The division instruction assumes your dividend is 128 bits wide, combining rdx (for the upper 64 bits) and rax (for the lower 64 bits) into rdx:rax. If rdx contains leftover garbage values from previous instructions, the CPU will try to divide a massive number and produce an overflow error or garbage results. That is why we must clear rdx first!
  2. Fixed Destination: After idiv rbx runs, the CPU automatically puts the quotient in rax and the remainder in rdx. You don't get a say in this; it's physically hardwired into the CPU.

Why xor rdx, rdx? XORing a register with itself is the standard, idiom-proven way to zero out a register in assembly. It is faster and results in a smaller binary size compared to writing mov rdx, 0 because the CPU executes register-to-register self-operations in a single clock cycle without loading an immediate value.

6. The CLI Calculator Program

To practice division, I was tasked with writing a calculator program (calc.asm) that evaluates the expression (20 + 4) / 3 and returns the remainder as the program's exit code, allowing us to inspect it using echo $? in the shell.

Here is the completed code:

section .text
    global _start

_start:
    mov rax, 20
    mov rbx, 4
    add rax, rbx      ; rax = 20 + 4 = 24

    mov rcx, 3
    xor rdx, rdx      ; clear rdx for division
    idiv rcx          ; rax = quotient (8), rdx = remainder (0)

    mov rax, 60
    mov rdi, rdx      ; exit code = remainder
    syscall

We built it and ran it:

$ nasm -f elf64 calc.o -o calc.asm
$ ld calc.o -o calc
$ ./calc
$ echo $?
0

Since 24 is perfectly divisible by 3, the remainder in rdx is 0, which matches the exit code! To make it more interesting, we changed the dividend from 20 to 25. Now the expression is (25 + 4) / 3 = 29 / 3. I predicted a quotient of 9 and a remainder of 2. We built and ran it again:

$ ./calc
$ echo $?
2

It worked flawlessly! The remainder was indeed 2.

7. Bitwise Logic Instructions

Beyond arithmetic, the CPU can manipulate values bit-by-bit. These instructions compare individual bits in parallel:

Instruction What it does Bit Logic Rule
and dst, src Bitwise AND 1 only if both bits are 1
or dst, src Bitwise OR 1 if either bit is 1
xor dst, src Bitwise Exclusive OR 1 if the bits are different
not dst Bitwise NOT Flips all bits (0 becomes 1, and vice-versa)
shl dst, count Shift Left Shifts bits left (effectively multiplies by 2count)
shr dst, count Shift Right Shifts bits right (effectively divides by 2count)

Why XORing to Zero Works (In Plain English)

Many assembly beginners get confused by xor rdx, rdx. To understand why it zeroes a register, look at the XOR rule: the output is 1 if and only if the inputs are different.

If you XOR a register with itself, you are comparing every bit with its own copy. A bit is never different from itself—a 1 matches a 1, and a 0 matches a 0. Because every single bit matches, every single comparison outputs 0. Consequently, the entire register is instantly cleared to zero. It is clean, elegant, and highly optimized by CPU hardware designers.

Three Practical Logic Use Cases

xor rax, rax        ; Zero out rax (faster than mov rax, 0)
and rax, 0xFF       ; Keep only the lowest byte (al), clearing the top 56 bits
shl rax, 1          ; Multiply rax by 2 (bit shifting is faster than imul!)

8. The Real Project: The ASCII Story

At the end of the lesson, my mentor gave me a new problem: how do you print a number to the screen?

In Lesson 1, we printed strings by pointing the write syscall at a memory address holding character bytes. But if rax = 9, we can't just pass rax to write. If we point the write syscall at an address containing the raw value 9, it will attempt to output the character with ASCII code 9 (which is a horizontal tab), not the character symbol "9"!

This led to a major "aha!" moment. The CPU has absolutely no concept of what the character "9" looks like. It does not know about symbols, fonts, or screens. It only knows numbers. To make the terminal print a visible 9, we have to send the numerical value 57—the ASCII code representing the character symbol '9'.

The translation rule is simple: since the ASCII code for '0' is 48, any single digit can be converted to its ASCII character value by adding 48:

ASCII character = digit + 48

To print the quotient of (25 + 4) / 3, we wrote a program that converts the division result to ASCII and saves it in a memory buffer before calling write:

section .data
    result db 0, 10        ; 1 byte for our ASCII character, 1 byte for newline (10)

section .text
    global _start

_start:
    mov rax, 25
    mov rbx, 4
    add rax, rbx      ; rax = 29

    mov rcx, 3
    xor rdx, rdx      ; zero rdx
    idiv rcx          ; rax = quotient (9), rdx = remainder (2)

    ; rax holds 9. Convert to ASCII: 9 + 48 = 57 ('9')
    add rax, 48
    mov [result], al  ; Store only the low-byte into our memory buffer!

    ; print our 2-byte result buffer (digit + newline)
    mov rax, 1
    mov rdi, 1
    mov rsi, result
    mov rdx, 2
    syscall

    ; exit with the remainder (rdx = 2) as the exit code
    mov rax, 60
    mov rdi, rdx
    syscall

Let's compile and run it:

$ nasm -f elf64 calc.asm -o calc.o
$ ld calc.o -o calc
$ ./calc
9
$ echo $?
2

Seeing that number 9 print out on the screen, followed by checking the exit code and seeing the remainder 2, felt incredibly satisfying. The program communicated two separate outputs simultaneously through two completely different channels!

Size Matching: Why We Used al

Notice the line mov [result], al. We stored the 8-bit register al instead of the full 64-bit rax. Why?

In our data section, we defined result db 0, 10. The db directive allocates exactly one byte for the digit and one byte for the newline character. If we wrote mov [result], rax, we would attempt to write 8 full bytes starting at the result address. This would overwrite our newline byte (10) and corrupt the next 7 bytes of adjacent memory. By using the low-byte register al, we write exactly 1 byte, matching the memory slot size perfectly. In assembly, always match your register size to your memory size!

Summary of Lesson 3:

  • Basic Arithmetic: add, sub, inc, dec, and imul modify registers in place.
  • Division Constraints: idiv uses rdx:rax as a 128-bit dividend. Always clear rdx (via xor rdx, rdx) first, and expect the quotient in rax and remainder in rdx.
  • The Zero Flag (ZF): The CPU silently sets ZF = 1 in the rflags register whenever a calculation results in zero. GDB allows us to trace this live.
  • Bitwise Operations: Instructions like and, or, xor, not, and bit shifts (shl, shr) manipulate individual bits.
  • XOR Self-Zeroing: XORing any value with itself results in 0 because no bit is ever different from itself.
  • ASCII Rule: Printing digits requires adding 48 to convert numerical values to their corresponding ASCII characters (e.g., 9 becomes 57).
  • Size Matching: Use appropriately sized sub-registers (like al for bytes) when moving data into allocated memory slots to avoid memory corruption.

Tracing values step-by-step, monitoring flags change under the hood, and converting calculations to screen characters showed me how math turns into logic, and logic turns into interface. Next lesson, we dive into loops and jumps, where we'll use these flags to build real control structures!
