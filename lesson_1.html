Lesson 1: From Zero to Hello World in NASM

Series: NASM Learning | Platform: Ubuntu/Linux x86-64

I've been comfortable writing Python and JavaScript for a while now. But honestly? I felt like I was just pressing buttons and watching things happen. I didn't know what the machine was actually doing underneath. So I decided to learn assembly — specifically NASM x86-64 on Linux. This is what I learned in week one, and how everything starts to connect.

What Actually Happens When You Run Code

Before writing a single line of assembly, I needed to understand one thing: what is a CPU actually doing?

It's simpler than you'd think. The CPU reads an instruction from memory, one at a time, and executes it. Then it reads the next one. Then the next. That's the entire loop — billions of times per second. Every program you've ever written — web frameworks, games, everything — is ultimately just that loop running your instructions.

NASM lets you write those instructions directly. No interpreter. No compiler making choices for you. You're talking straight to the CPU.

Three Things You Must Know First

1. Registers — The CPU's Scratch Pad

The CPU doesn't work on RAM directly. It has tiny, ultra-fast storage built into itself called registers. Think of them like variables that live inside the chip. In 64-bit mode, the main ones are named rax, rbx, rcx, rdx, rsi, rdi, rsp, rbp, and a few more. Each holds 64 bits (8 bytes) of data.

When you want to add two numbers, you load them into registers, add them there, then optionally move the result back to RAM.

2. Memory — The Big Slow Storage

RAM is where your program's code and data actually live. It's a giant array of bytes, each with an address. The CPU constantly reads from and writes to it, but registers are always the middle-man.

3. The Stack — Memory With a Discipline

The stack is a special region of memory that grows and shrinks as your program runs. It follows one rule: last in, first out. You push values onto it and pop them off. It's used for local variables, saving registers, and making function calls work.

Two registers are especially important:

  • rip: always points at the next instruction to execute. The CPU's job is read rip → run it → move rip forward → repeat.
  • rsp: always points at the top of the stack. It moves down (yes, down — the stack grows toward lower addresses) when you push, and up when you pop.

How Assembly Talks to the Kernel

Here's the key idea: assembly can't print text on its own. It can't exit. It has to ask the Linux kernel to do these things. This request is called a syscall — a system call.

Think of it like filling out a form. You put what you want in specific registers, call syscall, and the kernel reads that form and does the work.

The pattern is always the same:

  • rax = the syscall number (what service you want — like "is this the write service or the exit service?")
  • rdi, rsi, rdx = the arguments (the details for that service)
  • Then you execute syscall and the kernel handles it

For this lesson, two syscalls matter:

  • write (number 1): file descriptor in rdi (1 = screen), address in rsi, length in rdx
  • exit (number 60): exit code in rdi

Install NASM (All OS)

Use Ubuntu/Linux commands for this lesson. If you are on another OS, install NASM first then use your Linux environment for the build/run steps.

  • Ubuntu/Debian Linux: sudo apt update && sudo apt install nasm gcc
  • macOS (Homebrew): brew install nasm
  • Windows: download NASM from nasm.us, then use a Linux environment (WSL, Docker, or VM) for linking and running

Verify:

nasm --version
ld --version

Your First Program

Here it is. Before you run it, I want you to read it and think about what each line does:

section .data
    msg db "Hello, World!", 10
    len equ $ - msg

section .text
    global _start

_start:
    mov rax, 1
    mov rdi, 1
    mov rsi, msg
    mov rdx, len
    syscall

    mov rax, 60
    mov rdi, 0
    syscall

What's happening: .data section holds your data (the string). .text section holds your code. db means "define bytes" — NASM will write those characters (plus the newline, which is ASCII 10) into memory. len equ $ - msg is clever: $ means "current position" and msg is where the string started, so $ - msg calculates the byte length. No manual counting needed.

Build It

nasm -f elf64 hello.asm -o hello.o
ld hello.o -o hello
./hello

You should see:

Hello, World!

Now run this:

echo $?

You'll see 0. That's the exit code — whatever value you put in rdi before the exit syscall, the shell reports it afterward.

The Real Learning Happens Now

Here's where it started clicking for me. I changed the exit code:

    mov rax, 60
    mov rdi, 7
    syscall

Reassembled, relinked, ran it. Same output: Hello, World!. Then:

$ echo $?
7

That's when everything connected. The number was right there in the register. The shell just read it. Nothing magical. It's all just numbers flowing through registers.

Then I changed the message:

    msg db "Hello, Appu!", 10

Assembled, linked, ran it. The string printed. And I realized my mind had automatically kept asking: "Did I break the length calculation?" But I didn't — len equ $ - msg recalculated itself automatically at assemble time. The answer is built in.

Finally, I printed the message twice by duplicating the syscall block (no loops, just copy-paste):

_start:
    mov rax, 1
    mov rdi, 1
    mov rsi, msg
    mov rdx, len
    syscall

    mov rax, 1
    mov rdi, 1
    mov rsi, msg
    mov rdx, len
    syscall

    mov rax, 60
    mov rdi, 0
    syscall

Output: message printed twice. Still no magic — just instructions running one after another.
