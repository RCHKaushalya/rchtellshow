Lesson 2: Registers, Sub-registers, and the Power of Memory Brackets

Series: NASM Learning | Platform: Ubuntu/Linux x86-64

Lesson 1 was a thrill. Hearing the CPU execute raw machine instructions and seeing "Hello, World!" print to the screen made me feel closer to the hardware than ever. But as I sat down for Lesson 2, my mentor quickly showed me that we were barely scratching the surface of how the CPU actually stores and manipulates data. This lesson was full of "aha!" moments, simple mistakes on my part, and a deep-dive into the ultimate developer tool: the debugger.

1. The Russian Dolls Inside Your CPU

Up until now, I've been using registers like rax, rdi, rsi, and rdx as single, giant 64-bit storage containers. But here is the mind-bending reality: each register is actually four registers in one.

They aren't separate physical slots. Instead, they are different-sized windows looking into the exact same physical storage space. Let's look at rax as the primary example:

Register Name Size Which Bits It Accesses
rax 64-bit The whole physical register (8 bytes)
eax 32-bit The lower half (bits 0–31)
ax 16-bit The lower 16 bits (bits 0–15)
al 8-bit The lowest 8 bits (bits 0–7)

If rax holds the 64-bit hexadecimal value 0x0000000000000042, then looking through the smaller windows gives us: eax = 0x00000042, ax = 0x0042, and al = 0x42. It's the same physical silicon, just chopped up into smaller addressable units. This allows us to work on individual bytes without disturbing or wasting the rest of the register.

Mentor: "Let's put this to the test. If rax holds the value 0x00000000AABBCCDD, what would each sub-register contain? Reason it out."

Me: "Okay, let's see:

  • eax = 0xAABBCCDD (last 32 bit)
  • ax = 0xCCDD (last 8 bit)
  • al = 0xDD (last 4 bit)"

Mentor: "Almost perfect! The values are 100% correct, but your bit labels are mixed up. ax is 16 bits (2 bytes), and al is 8 bits (1 byte). Remember, two hexadecimal digits equal exactly one byte (8 bits). 0xDD is one full byte, not 4 bits!"

That correction immediately clarified things. A single hex digit is 4 bits (a nibble), so two hex digits (like DD) represent a full 8-bit byte.

My mentor also pointed out that there is a high-byte sub-register called ah, which accesses the upper byte of ax (the bits 8–15). For example, if ax = 0xCCDD, then ah = 0xCC and al = 0xDD. However, this is a legacy feature left over from the old 16-bit 8086 era. It only exists for the classic registers (rax, rbx, rcx, rdx). The newer registers added in later architectures (like rsi, rdi, or r8–r15) do not have an h variant—only low-byte windows!

2. The Workhorse of Assembly: mov

The most common instruction you'll write in assembly is mov. It copies data from a source to a destination. The syntax is:

mov destination, source

You have to train your brain to read this from right to left: "take the source, copy it into the destination." It can copy between registers, from memory to registers, from registers to memory, or load a literal number directly.

Important: mov is actually a copy instruction. It doesn't empty or clear the source register. After mov rbx, rax, both registers hold the identical value.

3. The Registers Experiment

To see this in action, we wrote a small, silent program called registers.asm:

section .data
    val dq 0x00000000AABBCCDD

section .text
    global _start

_start:
    mov rax, [val]
    mov rbx, rax

    mov rax, 60
    mov rdi, 0
    syscall

My mentor asked me three questions about this script that forced me to unpack what was happening:

Question 1: What does dq mean?
Knowing that db stands for "define byte" (1 byte), I guessed that dq meant "quarter word" storing 64 bits. My mentor corrected the term: it actually stands for define quadword. Here is the size hierarchy in x86-64 assembly:

  • db: Define Byte (1 byte / 8 bits)
  • dw: Define Word (2 bytes / 16 bits)
  • dd: Define Doubleword (4 bytes / 32 bits)
  • dq: Define Quadword (8 bytes / 64 bits) — fits a 64-bit register perfectly!

Question 2: Why does mov rax, [val] have brackets, but mov rax, 60 does not?
This is the golden rule of assembly. val is just a label representing a memory address (a number pointing to RAM).

  • Without brackets, val gives you the raw address itself.
  • With brackets, [val] dereferences the pointer. It means: "Go to the address called val, read the actual bytes stored there, and bring them back."
Meanwhile, 60 is just a literal number—no memory lookup needed.

4. Watching the CPU's Heartbeat with GDB

Running a program that exits silently without output is boring. To solve this, my mentor introduced me to the debugger: gdb. This lets us peek inside the CPU chip and watch the registers change in real-time as each line of assembly executes.

Here is how to set it up in WSL:

# Install GDB
sudo apt install gdb

# Assemble and Link with Debug Symbols (-g -F dwarf)
nasm -f elf64 -g -F dwarf registers.asm -o registers.o
ld registers.o -o registers

# Run the program under GDB
gdb ./registers

Inside the GDB prompt, I typed these commands:

(gdb) break _start
(gdb) run
(gdb) layout regs

Instantly, a gorgeous split-screen panel appeared showing all the CPU registers! I stepped through the instructions one-by-one by typing si (step instruction):

Me: "Wow, this is amazing. I can see everything changing line by line!"

Mentor: "What values did you see in GDB?"

Me: "After the first si (executing mov rax, [val]), rax immediately showed 0xaabbccdd 2864434397. And before the next instruction, rbx was sitting at 0x0 0, but after the next si, it became 0xaabbccdd 2864434397!"

Mentor: "Excellent. You can see GDB shows both the hex and decimal value of the register at once. To the CPU, they are identical. Also, notice that rbx started at exactly 0x0. The operating system actively zeroes out registers before passing control to your program for security reasons, so no leftover junk remains from other processes. Watching this in real-time shows you facts, not guesses."

5. The Subtle Art of Address Loading: LEA vs MOV

Next, we covered lea (Load Effective Address), which is often confused with mov.

  • mov rax, [val] goes to the address of val and copies the contents (data) stored there into rax.
  • lea rax, [val] loads the address (pointer) of val itself into rax.
In our Lesson 1 Hello World program, we did mov rsi, msg to pass the address of our string to the kernel. We could have written lea rsi, [msg] and it would produce the exact same instruction. It's all about whether we want the address or the contents.

6. Separating Code and Data: The Two-String Program

To cement these concepts, I wrote twostrings.asm, a program that prints my name (Rasindu) and the name of the assembler (NASM) on separate lines:

section .data
    name db "Rasindu", 10
    namelen equ $ - name
    lang db "NASM", 10
    langlen equ $ - lang

section .text
    global _start

_start:
    ; print name
    mov rax, 1
    mov rdi, 1
    mov rsi, name
    mov rdx, namelen
    syscall

    ; print lang
    mov rax, 1
    mov rdi, 1
    mov rsi, lang
    mov rdx, langlen
    syscall

    mov rax, 60
    mov rdi, 0
    syscall

I assembled and ran it, and it output:

Rasindu
NASM

My mentor challenged me to swap the print blocks so "NASM" printed first, and asked what needed to change. I swapped the syscall instructions in the .text section, left the .data section completely untouched, and it compiled and ran perfectly in the new order!

This proved a fundamental lesson: data and code are entirely independent. Data simply sits in memory. The code section decides where to look and what to do with it by loading the addresses we choose into registers.

We also analyzed the line langlen equ $ - lang. When NASM processes this, it has already written name (8 bytes: "Rasindu" + newline) into memory. When it reaches this line, the assembler counter symbol $ points exactly to the address immediately following the lang string. By subtracting lang from $, we get the precise length of the language string alone. Order in the data section matters because the assembler processes everything sequentially!

Summary of week two:

  • Sub-registers are windows: rax, eax, ax, and al let you touch 64, 32, 16, or 8 bits of the same physical storage.
  • Brackets mean lookups: [val] reads the data inside the address, while val is just the address number itself.
  • Size descriptors: dq (quadword = 8 bytes), dd (doubleword = 4 bytes), dw (word = 2 bytes), and db (byte = 1 byte) declare different allocations.
  • GDB reveals the truth: Using layout regs in the debugger lets you watch registers change instruction-by-instruction.
  • Data vs Code: Data stays put in RAM. Your instructions decide what to do with it, and in what order.

Moving from silent memory reads to live GDB tracing was the highlight of this week. Once you see the registers changing in front of your eyes, the magic fades, and you are left with pure, beautiful logic. Next up: Lesson 3 and arithmetic!
